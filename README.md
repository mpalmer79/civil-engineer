# Civil Engineer AI: Stormwater Review Assistant

Civil Engineer AI is a portfolio GenAI system that assists stormwater and
site-plan review using document evidence, checklist validation, risk flagging,
human review, audit logging, and evaluation tracking.

It is a **review-support and evidence-organization** system for land development
workflows. It is **not** a licensed engineering tool and does not replace
professional engineering judgment.

> **Professional boundary.** Civil Engineer AI assists review. It does not
> approve plans, certify compliance, stamp or seal drawings, replace a licensed
> Professional Engineer, or make final safety determinations. Every finding is a
> review-support issue that needs reviewer confirmation.

---

## Current phase: Phase 6, Plan Sheet and CAD-Aware Review Foundation

Phase 6 begins the transition from document-only review into plan sheet and
CAD-aware review support. It adds plan sheet intelligence and CAD-aware metadata
modeling, **not full CAD parsing**. No CAD file is parsed, no DWG or DXF data is
processed, and no Autodesk integration exists yet. It adds:

- A **plan sheet index** for the Brookside Meadows civil plan set
- A **CAD-aware metadata model** for civil features (basins, pipes, roads, lots,
  utilities, and more)
- **Plan references** that connect documents, sheets, and civil features
- **Missing sheet detection** (for example the referenced but not included
  revised grading sheet C-3.1)
- **Plan consistency findings** generated by a plan consistency check
- **Plan sheet and CAD review frontend pages**
- A **future Autodesk and CAD integration roadmap**
- Backend tests for sheet indexing, missing sheet detection, reference
  consistency, API endpoints, and safety language

Phase 6 keeps the professional boundary: there is no action called approve and
no status such as approved, certified, fully compliant, safe, CAD verified, or
design validated. CAD parsing is not implemented, Autodesk integration is not
implemented, no DWG or DXF files are processed, no CAD drawing is verified, and
no plan is approved. Every plan consistency finding requires human review.

Earlier phases established the product foundation:

- Phase 0: research, domain model, and the Brookside Meadows project story
- Phase 1: a static Next.js prototype driven by seed data
- Phase 2: a FastAPI backend, SQLite database, SQLAlchemy models, seeded data,
  API endpoints, tests, and frontend integration
- Phase 3: document chunks, finding sources, and keyword retrieval with
  evidence display
- Phase 4: a controlled AI review run workflow with a mock provider, evidence
  first prompts, strict JSON validation, safety and citation checks, and
  mandatory human review for every draft finding
- Phase 5: persisted human review actions with status transitions, and
  evaluation scoring against the expected findings with recall and precision
  metrics

The reviewed fixture remains **Brookside Meadows**: a 47-lot single-family
subdivision in the Town of Hartwell with a green-and-gray stormwater treatment
train and ten intentionally planted review issues.

## Getting started

### Frontend (Next.js)

```bash
npm install
npm run dev        # http://localhost:3000
npm run build
npm run typecheck
```

Set the backend URL for the frontend with an environment variable:

```bash
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

When the backend is reachable, pages render live data from it. When it is not,
they fall back to the local seed data so the app still works. Core project pages
include static fallback data, but Phase 3 source evidence and Phase 4 AI draft
findings are backend-canonical. If the backend is not running, evidence and AI
review panels show a status note instead of duplicated or stale data.

### Backend (FastAPI)

```bash
cd backend
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python -m app.db.seed            # load the Brookside Meadows fixture
uvicorn app.main:app --reload    # http://localhost:8000
pytest
```

See [`backend/README.md`](backend/README.md) for full backend setup and route
examples.

## What Phase 6 proves

- The product understands civil plan structure: it organizes the plan sheet
  index, connects sheets to documents, checklist items, and findings, and flags
  sheets that are referenced but not included.
- CAD-aware metadata is modeled before any CAD parsing: basins, pipes, roads,
  lots, and utilities are described as a future-ready abstraction that later
  phases can populate from real exports.
- Cross-references are reviewable: a plan consistency check compares plan sheets,
  references, and CAD-aware metadata and surfaces missing sheets and conflicting
  labels as review-support findings.
- The professional boundary holds: nothing is approved, certified, CAD verified,
  or design validated, and every plan consistency finding requires human review.
- The decision history is preserved: the plan consistency check writes audit
  events for each step.
