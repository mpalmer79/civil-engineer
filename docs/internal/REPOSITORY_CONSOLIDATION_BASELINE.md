## 9. Duplicate or overlapping modules

- Nine lib/api modules define private postJson/patchJson/deleteJson wrappers duplicating the shared client. Two competing result shapes exist (ApiResult for reads, an ad-hoc `{ ok, backendReachable }` object for mutations across roughly 20 modules).
- Two modules define local snake-to-camel mappers.
- Hand-written view-model types duplicate the generated OpenAPI schema, which no runtime code imports.
- Three project-dashboard page surfaces render the same seeded project.
- backend/app/core/safety.py mixes per-domain status vocabularies (60+ ALLOWED_* sets and transition maps) with prohibited-language guardrails and 6 helper functions; every bounded context imports it.
- 23 ad-hoc service *Error classes and roughly 197 HTTPException raises with per-router translation helpers; no shared exception handler.

## 11. Dependency graph pressure points

- app.core.safety: imported by nearly every backend service and router (shared vocabulary).
- app.db.models namespace: 106 import sites use `from app.db import models`; only models/__init__.py imports core.py directly, so redistributing model classes is mechanically safe if the package facade keeps re-exporting.
- lib/api/client.ts: shared read path for 37 API modules.
- Transaction ownership is diffuse: 144 session.commit() sites in services, 11 in routers, 7 in db/seed layers.
- Production code hard-depends on seeded reference-project identifiers: access_control_service.BROOKSIDE_PROJECT_ID, checklist_review_service.STARTER_RULE_PACK_ID, the `expected_status_for_brookside_meadows` checklist schema field, and cad sample_key default "brookside_meadows".

## 13. Known risks before refactoring

1. Splitting models/core.py must keep every class importable as models.X and keep Alembic metadata registration complete; a missed re-export silently drops tables from autogenerate comparisons.
2. safety.py is imported everywhere; renaming symbols breaks dozens of modules. Mitigation: split into a package that re-exports the existing names.
3. Documentation moves break the guide knowledge freshness gate (check:guide), recruiterDocs and releaseReadiness test suites, and verify scripts unless updated together.
4. The vitest 2 to 4 migration can change coverage measurement enough to move ratcheted thresholds in either direction.
5. The fastapi/starlette upgrade needed for the Python audit can change validation behavior and response details that tests assert on.
6. The DXF proof artifacts are byte-deterministic and CI-gated; any change to the parse pipeline must keep regenerated artifacts identical or regenerate and commit them together.
7. The frontend content-integrity gate forbids em dashes and attribution strings across the repo; all new documents must comply.
8. Seeded-identifier coupling means reordering seed execution or renaming seed constants can break startup and tests.
