# ADR 0005: Backend domain decomposition

Status: Proposed (transitional architecture in force)

## Context

`backend/app/db/models.py` holds roughly 3,000 lines covering every domain,
and several services exceed 1,500 lines (`cad_intake_service`,
`review_cycle_service`, `command_center_service`,
`evidence_retrieval_service`). The code is organized, tested (about 93 percent
coverage), and stable, but a single model module is the only map of the data
domain, and the largest services mix querying, validation, persistence,
mapping, and audit emission.

## Decision

Target domain boundaries, derived from the existing service and route
structure:

- Identity and organizations (auth, memberships, invitations)
- Projects and access control
- Documents and storage
- PDF evidence (pages, chunks, citations, retrieval)
- CAD intake (DXF files, parse runs, plan-sheet comparison)
- Checklists and rule packs
- Findings and human review
- Review cycles and resubmittals
- Applicant responses and response packages
- Review packets and workflow
- Command center and dashboards
- Audit
- Billing and usage

Decomposition proceeds domain by domain:

1. Split `models.py` into `app/db/models/<domain>.py` modules re-exported
   through `app/db/models/__init__.py`, preserving every table name, column,
   constraint, relationship, and the migration history. The compatibility
   re-export keeps `from app.db import models` working unchanged, so no
   migration is generated by the move.
2. Extract the largest services along responsibility seams (query
   orchestration, state transitions, mapping, audit emission) while keeping
   the public service functions stable, one domain per change, each behind the
   existing test suite and the 90 percent coverage gate.

## Transitional architecture

Until a domain is extracted, its behavior is protected by the current tests:
787 backend tests exercise the services through the API surface, and the
coverage gate prevents silent regression. Route modules already avoid business
logic (they parse requests, resolve auth context, and delegate to services),
and state transitions for packages and review cycles are centralized in their
services. This ADR records the target so extraction happens deliberately, not
as a big-bang rewrite.
